# 聚合权益管理后台｜登录与权限系统后端交付文档

## 1. 文档定位

本目录是一套可直接交给 Codex 执行的后端实施文档包，范围**严格限定**在你提供的需求文档内，不扩展任何额外业务模块。

当前仅覆盖以下 6 个业务域：

1. 登录认证
2. 员工管理
3. 主体配置
4. 用户组与授权
5. 阿里云短信配置
6. 操作日志 / 登录日志

## 2. 已固定的技术决策

### 2.1 项目栈

- Go
- GoFrame v2
- MySQL
- Redis
- JWT
- bcrypt

### 2.2 鉴权方案

采用 **JWT + Redis 会话态 + token_version** 组合方案，原因如下：

- 仅 JWT 无法满足“删除员工后 Token 立即失效”
- 仅 Redis Session 不利于前后端 Bearer Token 统一
- `token_version` 可以一次性让某个用户的所有旧 Token 失效
- Redis 记录每个 `jti` 会话，支持退出登录与精确失效

### 2.3 权限方案

采用 **用户组授权 + 菜单权限码 + 中间件统一校验**：

- 员工不直接挂权限，只绑定用户组
- 超管 `group_id = 0`，不受权限校验约束
- 普通员工通过 `admin_group_menu -> admin_menu.code` 获取权限码
- 每个接口在 `api` 层写清楚 `permission` / `superOnly` / `noAuth` 元数据
- 权限校验在统一中间件中完成

### 2.4 项目形态

采用 **GoFrame v2 大型单体项目** 组织方式，输出目录按以下主干执行：

- `api`
- `internal`
- `cmd`
- `manifest`
- `hack`
- `resource`
- `utility`

> 说明：GoFrame 官方目录设计允许按项目复杂度增减默认目录；本方案在保留 `api / internal / manifest / utility` 核心思路的前提下，将启动入口放入根级 `cmd`，方便大型项目启动与部署脚本管理。

## 3. 需求文档中的冲突点，已在技术文档中修正

### 3.1 员工添加/编辑接口缺少 `phone`

需求页面字段中，员工“添加/编辑”都明确要求手机号，但接口示例没有写 `phone`。  
本交付文档统一按**字段规则优先**处理：

- `POST /api/admin/user/add` 必填 `phone`
- `PUT /api/admin/user/:id` 必填 `phone`

### 3.2 权限码清单缺少部分中文显示名

需求文档给出了完整权限码，但没有给全所有二级/三级菜单中文名称。  
为避免瞎猜，本交付文档中的 `admin_menu` 初始化策略为：

- 已明确给出中文名称的，按中文写入 `name`
- 未明确给出中文名称的，`name` 暂以 **code 同名** 写入
- 后续如果你补齐正式菜单中文文案，Codex 再批量更新 `admin_menu.name`

## 5. 给 Codex 的总执行顺序

1. 先落目录结构与基础配置
2. 再执行数据库建表与 seed
3. 再生成 dao/do/entity
4. 再按 phase-1 到 phase-4 顺序逐期实现
5. 每一期完成后先跑自测，再进入下一期
6. 任何单文件超过约定体量必须立即拆分，不允许出现几千行巨型文件

## 6. 文件体量硬约束

- controller 单文件建议不超过 **200 行**
- logic 单文件建议不超过 **300 行**
- service interface 单文件建议不超过 **200 行**
- middleware 单文件建议不超过 **200 行**
- consts / dto / vo / converter 按领域拆分，不做“大杂烩”

超过体量必须拆分为：

- `login.go`
- `login_sms.go`
- `login_token.go`
- `logout.go`
- `me.go`

这种“按用例拆”的细粒度文件，而不是把整模块全部塞进一个文件。

## 7. 注释硬约束

必须写注释的位置：

- 导出的结构体 / 接口 / 方法
- 复杂业务规则判断分支
- Token、Redis Key、验证码、权限缓存等关键流程
- 事务边界
- 任何“为什么这么做”而不是“代码做了什么”的地方

不要写废话注释；但复杂规则必须写清楚原因与边界。
