## 2. 各目录职责

### 2.1 `api`

只放接口定义，不写业务实现。  
每个接口一个文件，避免一个 `api.go` 塞几十个请求对象。

#### 要求

- 一个接口文件里最多只放一组强关联 `Req/Res`
- 每个请求对象都写 `g.Meta`
- 在 `g.Meta` 中声明权限元数据
- 在字段标签中写清楚校验规则与错误文案

### 2.2 `internal/controller`

按业务模块拆目录，再按接口拆文件。

#### 禁止

- 直接操作 DAO
- controller 中起事务
- 在 controller 中拼装复杂权限/缓存逻辑

### 2.3 `internal/service`

只保留接口。  
每个模块一个 interface 文件，不要把所有 service 接口塞在一起。

### 2.4 `internal/logic`

核心业务实现目录。  
规则：**对象一个、文件多拆**。

示例：

```text
internal/logic/admin/auth/
├── auth.go                # 注册 service
├── login.go               # 第一步登录
├── login_sms_send.go      # 发送验证码
├── login_sms_verify.go    # 验证验证码
├── logout.go              # 退出登录
└── me.go                  # 当前用户信息
```

### 2.5 `internal/dao`

全部通过 `gf gen dao` 生成。  
原则：

- 生成文件不手改
- 自定义查询写在 logic 中组合
- 需要复杂 SQL 时，优先在 logic 中使用 dao.Model()

### 2.6 `internal/model`

#### `entity`

数据库实体，工具生成。

#### `do`

数据库写入/更新对象，工具生成。

#### `dto`

手写业务输入输出对象，用于 logic / service 层。

#### `contextx`

当前登录用户上下文结构。

### 2.7 `internal/middleware`

统一放横切逻辑：

- JWT 鉴权
- 权限校验
- 统一响应
- panic 恢复
- 请求日志

### 2.8 `utility`

放与业务无关、可复用、无数据库依赖的小工具。

### 2.9 `manifest`

- 配置文件
- SQL 脚本
- 可部署物料

### 2.10 `hack`

- 本地辅助脚本
- 代码生成脚本
- lint / test / seed 脚本

## 4. 单文件体量红线

| 类型 | 建议上限 | 超限后的拆分方式 |
|------|---------|----------------|
| api 文件 | 120 行 | 一个接口一个文件 |
| controller 文件 | 200 行 | 一个方法一个文件 |
| logic 文件 | 300 行 | 按用例拆分 |
| middleware 文件 | 200 行 | JWT / 权限 / 响应 / recover 分开 |
| dto 文件 | 150 行 | 按子场景拆分 |

> 绝对禁止出现 `auth.go` 1000+ 行、`user.go` 2000+ 行、`controller.go` 3000+ 行这种文件。

## 5. 注释规范

### 5.1 必写注释

#### 导出接口

```go
// IAuth 定义登录认证领域服务。
type IAuth interface {
    // Login 执行账号密码登录第一步。
    // 当需要短信二次验证时，仅返回 login_token，不签发正式 token。
    Login(ctx context.Context, in dto.AdminLoginInput) (out dto.AdminLoginOutput, err error)
}
```

#### 复杂分支

```go
// 被删除账号对外统一返回“账号或密码错误”，避免暴露账号真实状态。
if user.IsDeleted == 1 {
    return nil, gerror.NewCode(gcode.CodeBusinessValidationFailed, "账号或密码错误")
}
```

#### Redis Key 设计

```go
// admin:login:tmp:{login_token}
// 存储登录第二步的临时状态，TTL 5 分钟。
// 仅在首次登录/IP 变更时生成。
```

### 5.2 不要写的注释

```go
// 给变量赋值
name := req.Name
```

这种废话注释不要出现。

## 6. 命名规范

### 6.1 文件命名

- 全小写
- 下划线分词
- 按动作命名

正确示例：

- `login_sms_verify.go`
- `batch_business.go`
- `auth_save.go`

错误示例：

- `userManager.go`
- `UserList.go`
- `common.go`（没有实际语义）

### 6.2 结构体命名

- API 请求：`LoginReq`
- API 响应：`LoginRes`
- DTO 输入：`AdminLoginInput`
- DTO 输出：`AdminLoginOutput`
- Logic 对象：`sAuth`

### 6.3 service 接口命名

- `IAuth`
- `IUser`
- `IGroup`
- `IConfig`
- `ILog`

## 7. 代码生成约束

### 7.1 dao/do/entity

统一通过脚本执行：

```bash
gf gen dao
```

### 7.2 controller

可以使用生成命令辅助创建，但生成后必须按本文档二次拆分：

- 不允许生成一个超大 controller 后直接堆逻辑
- 生成只是脚手架，不是最终结构

## 8. 测试文件建议

每个 logic 模块至少应保留：

```text
internal/logic/admin/auth/
├── login_test.go
├── login_sms_verify_test.go
└── logout_test.go
```

员工、用户组、配置模块同理。

## 9. Codex 执行要求

Codex 实施时必须遵守：

1. 先建空目录与占位文件
2. 再生成 dao/do/entity
3. 再补 service
4. 再补 logic
5. 最后写 controller 与 route
6. 每完成一个模块立即跑最小可执行自测
