# 05｜全局开发约束与代码规范

## 1. 总体原则

### 1.1 不扩展业务模块

当前代码只允许实现：

- 登录认证
- 员工管理
- 主体配置
- 用户组与授权
- 阿里云短信配置
- 操作日志
- 登录日志

不得顺手新增：

- 客户管理
- 订单管理
- 商品管理
- 财务管理
- 通知中心
- 文件上传中心

这些都不在本期范围内。

### 1.2 不允许大而全文件

任何时候都不能为了省事把：

- 所有接口写进一个 controller
- 所有逻辑写进一个 logic
- 所有常量写进一个 consts
- 所有 DTO 写进一个 dto.go

## 2. API 元数据约定

### 2.1 标准写法

```go
type ListReq struct {
    g.Meta `path:"/user/list" method:"get" tags:"员工管理" summary:"员工列表" permission:"admin.list"`
    Page     int `p:"page" d:"1" v:"min:1#页码最小为1"`
    PageSize int `p:"page_size" d:"20" v:"between:1,100#每页条数范围错误"`
}
```

### 2.2 自定义 Meta 标签

| 标签 | 作用 | 说明 |
|------|------|------|
| `noAuth:"true"` | 无需登录 | 登录、验证码发送、验证码验证 |
| `permission:"xxx"` | 需要权限码 | 普通员工接口 |
| `superOnly:"true"` | 仅超级管理员 | 短信配置保存/查询 |
| `tags:"..."` | 接口分组 | 便于生成文档 |
| `summary:"..."` | 接口摘要 | 便于审查 |

### 2.3 中间件判定规则

执行顺序：

1. 如果 `noAuth=true`，跳过 JWT 与权限检查
2. 否则必须先校验 JWT
3. `superOnly=true` 时必须 `group_id=0`
4. `permission` 为空时，只做登录校验，不做菜单权限校验
5. `permission` 不为空时，超管直接放行，普通员工校验权限码

## 3. 输入校验规范

### 3.1 用户名

规则：

- 字母开头
- 字母 + 数字
- 长度 6~10

正则：

```text
^[A-Za-z][A-Za-z0-9]{5,9}$
```

### 3.2 密码

规则：

- 字母开头
- 长度 6~10
- 仅允许字母、数字、下划线

正则：

```text
^[A-Za-z][A-Za-z0-9_]{5,9}$
```

### 3.3 手机号

正则：

```text
^1\d{10}$
```

### 3.4 短信验证码

正则：

```text
^\d{6}$
```

## 4. 统一返回规范

### 4.1 成功

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

### 4.2 失败

```json
{
  "code": 400,
  "msg": "参数错误",
  "data": null
}
```

### 4.3 分页返回

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "list": [],
    "pagination": {
      "page": 1,
      "page_size": 20,
      "total": 0
    }
  }
}
```

## 5. 错误文案规范

### 5.1 登录场景

- 账号不存在 → `账号或密码错误`
- 密码错误 → `账号或密码错误`
- 已删除账号 → `账号或密码错误`
- 账号禁用 → `账号已被禁用，请联系管理员`
- 未配置手机号 → `请联系管理员配置手机号`
- 验证码错误 → `验证码错误，剩余 N 次机会`
- 验证码发送过快 → `请稍后再试`

### 5.2 员工管理

- 用户名重复 → `用户名已存在`
- 手机号格式错误 → `手机号格式错误`
- 恢复用户名冲突 → `用户名已被占用，无法恢复`
- 超管禁止删除 → `超级管理员不允许删除`

### 5.3 用户组

- 用户组名重复 → `用户组名称已存在`
- 用户组下仍有员工 → `该用户组下还有 N 名员工，请先转移`

### 5.4 主体

- 主体名称重复 → `主体名称已存在`

## 6. Redis Key 规范

### 6.1 会话类

```text
admin:session:{jti}
admin:user:sessions:{user_id}
admin:login:tmp:{login_token}
```

### 6.2 短信类

```text
sms:login:{user_id}
sms:login:send_lock:{user_id}
```

### 6.3 权限与配置类

```text
admin:perm:group:{group_id}
admin:config:sms
```

### 6.4 Key 编写要求

- 全小写
- 业务前缀明确
- 不允许出现 `tmp1`、`abc`、`cache` 这种模糊命名
- Key 必须在 `internal/consts/cache_key.go` 集中声明

## 7. 事务与并发规范

### 7.1 必须使用事务的场景

- 删除员工
- 恢复员工
- 保存用户组授权
- 保存短信配置（批量 upsert）

### 7.2 不在事务里做的事情

以下动作应在事务提交后执行：

- 删除 Redis Session
- 删除权限缓存
- 投递异步操作日志
- 调短信外部服务

### 7.3 并发修改保护

对以下场景必须重新查询并判定最新状态：

- 编辑员工
- 删除员工
- 恢复员工
- 修改用户组状态
- 保存授权

避免“页面拿到旧数据，后端直接覆盖”的问题。

## 8. 日志规范

### 8.1 操作日志描述格式

必须是**中文业务描述**，并带关键参数。

正确示例：

- `添加员工：admin02，用户组：客服`
- `编辑员工：admin02，手机号：13800001111，用户组：运营`
- `删除员工：admin02`
- `恢复员工：admin02`
- `保存用户组授权：客服，共 18 个权限`
- `更新短信配置：签名=玖权益，模板=SMS_308586082`

错误示例：

- `update user`
- `success`
- `修改了配置`
- `编辑操作`

### 8.2 登录日志

每次登录成功后写入：

- admin_id
- admin_name
- ip
- ip_region
- created_at

### 8.3 请求日志

可以保留基础访问日志，但不要把完整密码、验证码、AccessKeySecret 打进日志。

## 9. 密码与敏感信息规范

### 9.1 密码

- 只允许 bcrypt 存储
- 不允许任何明文回显
- 不允许打印到日志
- 编辑员工不填密码时，绝不覆盖原哈希

### 9.2 AccessKeySecret

- 入库明文（便于调用短信服务），但查询时必须脱敏
- 脱敏格式：前 4 位 + `****`
- 不能出现在操作日志详情里；操作日志只记录配置项名称，不记录真实密钥

### 9.3 手机号

- 登录第一步返回的手机号必须脱敏显示
- 其他后台接口如需返回员工手机号，可按业务直接返回，不额外脱敏

## 10. 查询规范

### 10.1 列表分页

默认：

- `page = 1`
- `page_size = 20`

最大值建议：

- `page_size <= 100`

### 10.2 排序

默认全部按 `id DESC` 或 `created_at DESC`，以最新数据优先。

### 10.3 模糊搜索

- 操作日志按 `description LIKE '%keyword%'`
- 关键字为空时不拼装 LIKE 条件

## 11. 软删除规范

仅 `admin_user` 做软删除：

- `is_deleted = 1`
- `deleted_at = NOW()`
- `status = 0`
- `username = 原用户名_时间戳`

恢复时：

- `is_deleted = 0`
- `deleted_at = NULL`
- `status = 0`
- 用户名恢复为原用户名
- 若冲突则恢复失败

## 12. Service 注册规范

示例：

```go
// internal/service/auth.go
type IAuth interface {
    Login(ctx context.Context, in dto.AdminLoginInput) (out dto.AdminLoginOutput, err error)
}

var localAuth IAuth

func RegisterAuth(i IAuth) {
    localAuth = i
}

func Auth() IAuth {
    if localAuth == nil {
        panic("service IAuth not registered")
    }
    return localAuth
}
```

```go
// internal/logic/admin/auth/auth.go
type sAuth struct{}

func init() {
    service.RegisterAuth(New())
}

func New() *sAuth {
    return &sAuth{}
}
```

## 14. 测试最低要求

### 14.1 必须覆盖

- 登录成功
- 首次登录需要短信
- IP 变更需要短信
- 验证码错误次数耗尽
- 退出登录失效
- 删除员工失效
- 恢复员工冲突
- 用户组删除拦截
- 授权保存后权限生效
- 短信配置脱敏返回

### 14.2 建议覆盖

- 缓存回源
- 权限缓存刷新
- 操作日志队列满时降级同步写入

## 15. 禁止事项

- 禁止 controller 直接写 SQL
- 禁止 service interface 写实现
- 禁止把密码、验证码、AccessKeySecret 打日志
- 禁止用一个 `common.go` 承载所有通用逻辑
- 禁止直接返回数据库错误给前端
- 禁止在未确认需求的情况下扩业务模块
