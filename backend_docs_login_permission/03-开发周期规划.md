# 03｜开发周期规划

## 1. 分期原则

本项目虽然需求点不算特别多，但鉴权、权限、日志、软删除、短信验证之间存在强依赖，不能按页面顺序硬切。  
因此按“**先基础设施，再业务 CRUD，再授权，再配置与审计**”的顺序拆成 4 期。

## 2. 四期总表

| 周期 | 目标 | 覆盖需求 | 交付重点 | 退出标准 |
|------|------|---------|---------|---------|
| Phase 1 | 跑通项目骨架与完整登录链路 | F01~F03 | GoFrame 骨架、JWT+Redis、短信二步登录、`/me`、`/logout`、登录日志写入 | 能登录、能退出、权限中间件可工作 |
| Phase 2 | 完成员工与主体业务闭环 | F04~F14 | 员工列表/新增/编辑/删除/禁用/恢复/通知/商务标记、主体配置 | 员工与主体模块全可用，删除与恢复闭环 |
| Phase 3 | 完成用户组与授权系统 | F15~F17 | 用户组 CRUD、权限树、授权保存、权限缓存刷新 | 授权后普通员工下次请求即可生效 |
| Phase 4 | 完成短信配置与审计能力 | F18~F20 | 阿里云短信配置、操作日志列表、登录日志列表、异步操作日志 | 配置可保存，日志可查询，关键操作可审计 |

## 3. Phase 1｜基础骨架与登录鉴权

### 3.1 范围

- 项目目录落地
- 配置文件
- MySQL / Redis 接入
- 统一响应与错误码
- JWT 工具
- Redis Session 管理
- 账号密码登录
- 首次登录 / IP 变更短信二次验证
- `/api/admin/me`
- `/api/admin/logout`
- 登录日志写入
- JWT 中间件
- 权限中间件骨架

### 3.2 依赖

- 先执行建表 SQL
- 先写入超管数据
- 先写入菜单 seed
- 短信配置默认 seed 必须先写

### 3.3 交付成果

- 可启动服务
- 可使用超管账号登录
- 可返回 token / 当前用户信息 / 权限码
- 可对无效 token 返回 401
- 可对禁用账号返回 403
- 可对被删除账号统一返回账号密码错误
- 可记录登录日志

### 3.4 验收用例

1. 正确账号密码 + 不需要短信 → 直接登录成功
2. 正确账号密码 + 需要短信 → 返回 `login_token`
3. 错误验证码累计 5 次 → `login_token` 作废
4. 退出登录后旧 token 立即失效
5. 删除员工后旧 token 立即失效
6. 禁用员工后下次请求返回 403

## 4. Phase 2｜员工管理与主体配置

### 4.1 范围

- 员工列表
- 添加员工
- 编辑员工
- 删除员工（软删除）
- 状态切换
- 余额阈值通知切换
- 批量设置/取消商务
- 回收站列表
- 恢复员工
- 主体配置列表
- 添加主体
- 编辑主体

### 4.2 依赖

- Phase 1 鉴权已完成
- 用户组基础表可用
- 操作日志写接口可以先留空调用，正式异步化放到 Phase 4 完成

### 4.3 交付成果

- 员工 CRUD 与状态流转闭环
- 删除后进入回收站
- 恢复后状态默认禁用
- 主体名称唯一校验
- 操作日志调用点已埋好

### 4.4 验收用例

1. 用户名唯一校验生效
2. 用户名只读不可编辑
3. 删除后用户名自动追加后缀
4. 恢复时去后缀并校验冲突
5. 恢复后状态默认为禁用
6. 主体不允许删除

## 5. Phase 3｜用户组与权限授权

### 5.1 范围

- 用户组列表
- 添加用户组
- 编辑用户组
- 删除用户组
- 用户组状态切换
- 权限菜单树
- 获取用户组已授权菜单
- 保存授权
- 权限缓存刷新

### 5.2 依赖

- `admin_menu` seed 已完成
- Phase 1 权限中间件骨架已完成
- Phase 2 员工模块可用，用于删除用户组前做人数校验

### 5.3 交付成果

- 授权树可正常渲染
- 保存授权后普通员工下次请求即可拿到最新权限
- 用户组删除前能阻止存在员工的情况
- 超管不受权限限制

### 5.4 验收用例

1. 新建用户组后可授权
2. 授权保存后接口权限立即生效
3. 删除带员工的用户组会被拦截
4. 禁用用户组后组内员工下次请求返回 403 或无权限
5. `/api/admin/menu/tree` 返回树形结构正确

## 6. Phase 4｜短信配置与审计日志

### 6.1 范围

- 获取短信配置
- 保存短信配置
- AccessKeySecret 脱敏返回
- 操作日志列表
- 登录日志列表
- 异步操作日志写入 worker
- IP 归属地离线解析接入点

### 6.2 依赖

- Phase 1 登录与短信发送链路已完成
- Phase 2/3 写操作调用点已预留
- `resource/ipdb` 离线 IP 库文件已放入

### 6.3 交付成果

- 超管可维护短信配置
- 普通员工不能访问短信配置接口
- 关键操作均可落操作日志
- 可按人员 / 关键词 / 时间范围查询操作日志
- 可按人员 / 时间范围查询登录日志

### 6.4 验收用例

1. 短信配置保存成功后立即生效
2. AccessKeySecret 查询时脱敏
3. 操作日志默认倒序分页
4. 登录日志默认倒序分页
5. IP 解析失败时不影响登录成功

## 7. 每期统一开发顺序

每一期都按以下顺序执行，不允许乱序：

1. 新增/调整 API 对象
2. 新增 service 接口
3. 新增/调整 logic
4. 新增 DAO 查询组合
5. 新增 controller 方法
6. 注册 route
7. 写单元测试 / 接口自测
8. 更新操作日志写入点
9. 更新阶段文档状态

## 8. 每期统一输出清单

每一期至少输出以下结果：

- 代码目录变更清单
- 接口清单
- 表与索引变更清单
- Redis Key 变更清单
- 测试用例清单
- 已知风险与回归点

## 9. 给 Codex 的执行边界

Codex 必须按 Phase 顺序实现，不允许：

- 先写后期日志列表，再回头补登录
- 先硬编码权限，再后补数据库授权
- 先把所有逻辑写在 controller，再回头重构
- 为了省事把多个模块塞进一个 logic 文件

## 10. 里程碑检查点

### 10.1 Phase 1 结束后必须能做到

- 启动服务
- 登录
- 获取当前用户
- 退出登录
- 中间件鉴权拦截

### 10.2 Phase 2 结束后必须能做到

- 管理员列表与回收站可用
- 软删除与恢复闭环
- 主体配置可用

### 10.3 Phase 3 结束后必须能做到

- 用户组授权可用
- 权限随授权变更而生效

### 10.4 Phase 4 结束后必须能做到

- 短信配置可维护
- 审计查询可用
- 关键写操作可追踪
