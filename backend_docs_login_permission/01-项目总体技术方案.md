# 01｜项目总体技术方案

## 1. 目标与范围

本方案只实现以下模块，不新增任何业务模块：

- 登录认证
- 员工管理
- 主体配置
- 用户组与授权
- 阿里云短信配置
- 操作日志
- 登录日志

## 2. 架构总览

### 2.1 分层设计

```text
api（请求/响应定义，含接口元数据）
    ↓
controller（接收请求，调用 service）
    ↓
service（领域接口，禁止写实现）
    ↓
logic（业务实现）
    ↓
dao / model(do, entity)（数据访问）
    ↓
MySQL / Redis / 外部短信服务
```

### 2.2 分层职责

#### api 层

职责：

- 定义接口入参/出参
- 使用 `g.Meta` 声明 path / method / summary / tags
- 声明 `noAuth` / `permission` / `superOnly` 等鉴权元数据
- 做基础参数校验，不写业务逻辑

#### controller 层

职责：

- 接收 `api` 请求对象
- 调用 `service`
- 将 `service` 结果映射为 API 返回结构
- 不直接访问 `dao`
- 不写事务

#### service 层

职责：

- 仅保留接口定义
- 解耦 controller 与 logic
- 方便单测 mock

#### logic 层

职责：

- 业务规则实现
- 事务控制
- 调用 dao / redis / 外部适配器
- 触发操作日志、登录日志、缓存刷新、token 失效

#### dao / model 层

职责：

- DAO 只负责数据表访问
- `entity` 只表示数据库实体
- `do` 只表示写库/查库条件
- 不在 DAO 中拼装复杂业务返回对象

## 3. 核心技术决策

### 3.1 登录与会话模型

采用“双层鉴权”：

1. **JWT** 作为前端 Bearer Token
2. **Redis Session** 作为服务端会话状态

JWT claims 必须至少包含：

- `user_id`
- `group_id`
- `token_version`
- `jti`
- `exp`

Redis 维护：

- `admin:session:{jti}` → 当前 token 会话
- `admin:user:sessions:{user_id}` → 当前用户活跃 jti 集合
- `admin:login:tmp:{login_token}` → 登录第二步临时凭证
- `sms:login:{user_id}` → 登录验证码
- `sms:login:send_lock:{user_id}` → 发送间隔控制
- `admin:perm:group:{group_id}` → 用户组权限码缓存
- `admin:config:sms` → 短信配置缓存

### 3.2 立即失效机制

以下场景必须立即让旧 Token 失效：

- 退出登录
- 删除员工
- 禁用员工
- 恢复员工后重新启用前
- 密码重置（建议一并失效）

实现方式：

1. Token 中带 `token_version`
2. `admin_user.token_version` 为服务端真值
3. 中间件校验 token 中的 `token_version == db.token_version`
4. 退出登录时删除对应 `jti` session
5. 删除/禁用/改密时：
   - `token_version = token_version + 1`
   - 删除该用户 `admin:user:sessions:{user_id}` 下全部 `jti` session

### 3.3 权限模型

```text
admin_user (N:1) admin_group
admin_group (N:M) admin_menu
```

约束：

- 员工不直接挂权限
- 权限只通过用户组分配
- 超管 `group_id = 0`
- 普通员工权限集来源于 `admin_group_menu`

### 3.4 权限校验方式

使用 **接口元数据 + 统一中间件**：

- `noAuth:"true"`：登录、验证码发送、验证码验证等无需登录
- `superOnly:"true"`：仅超级管理员可访问
- `permission:"admin.list"`：普通鉴权接口

示例：

```go
type ListReq struct {
    g.Meta `path:"/user/list" method:"get" tags:"员工管理" summary:"员工列表" permission:"admin.list"`
    Page     int `p:"page" d:"1" v:"min:1#页码最小为1"`
    PageSize int `p:"page_size" d:"20" v:"between:1,100#每页条数范围 1~100"`
}
```

中间件执行顺序：

```text
请求进入
  ↓
Recover 中间件
  ↓
统一响应封装中间件
  ↓
JWT 中间件
  ↓
权限中间件
  ↓
controller
```

### 3.5 短信验证码登录模型

第一步 `/api/admin/login`：

- 校验账号密码
- 校验状态 / 删除状态
- 判断是否首次登录或 IP 变化
- 不需要短信：直接签发正式 token
- 需要短信：生成 `login_token` 写 Redis，返回 `need_sms_verify=true`

第二步 `/api/admin/login/sms/send`：

- 校验 `login_token`
- 校验发送间隔
- 读取短信配置
- 生成 6 位验证码写 Redis
- 调阿里云短信发送

第三步 `/api/admin/login/sms/verify`：

- 校验 `login_token`
- 校验验证码
- 错误次数 +1
- 成功后删除验证码
- 更新 `last_login_ip`
- 记录登录日志
- 签发正式 token

### 3.6 日志模型

#### 登录日志

触发时机：

- 每次登录成功后

字段：

- admin_id
- admin_name（快照）
- ip
- ip_region
- created_at

#### 操作日志

触发时机：

- 员工、主体、用户组、授权、短信配置等关键写操作

字段：

- admin_id
- admin_name（快照）
- description
- ip
- created_at

写入方式：

- 使用内存队列 + 后台 worker 异步写入
- 队列满时降级为同步写库并记录警告日志，避免日志丢失

## 4. 请求响应约定

### 4.1 统一响应结构

```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

### 4.2 统一错误码建议

| 场景 | code | 说明 |
|------|------|------|
| success | 0 | 成功 |
| 参数错误 | 400 | 请求参数不合法 |
| 未登录/登录过期 | 401 | Token 无效或过期 |
| 无权限/账号禁用 | 403 | 权限不足或账号状态异常 |
| 资源不存在 | 404 | 数据不存在 |
| 业务冲突 | 409 | 用户名冲突、主体名冲突等 |
| 发送频繁 | 429 | 验证码发送过快 |
| 系统错误 | 500 | 未分类内部错误 |

## 5. 缓存策略

### 5.1 用户组权限缓存

缓存键：`admin:perm:group:{group_id}`

缓存内容：

```json
["admin.list", "admin.department", "admin.action"]
```

缓存刷新时机：

- 修改用户组授权
- 修改用户组状态
- 删除用户组
- 用户登录时懒加载
- `/me` 请求时可复用缓存

### 5.2 短信配置缓存

缓存键：`admin:config:sms`

缓存刷新时机：

- 更新短信配置后立即删除缓存
- 下次读取时回源数据库

### 5.3 会话缓存

缓存键：

- `admin:session:{jti}`
- `admin:user:sessions:{user_id}`

刷新时机：

- 登录创建
- 退出删除
- 删除/禁用/改密批量删除

## 6. 事务边界

必须显式事务的场景：

1. 删除员工
   - 更新 `admin_user`
   - 递增 `token_version`
   - 提交后再删 Redis Session

2. 恢复员工
   - 恢复用户名
   - `is_deleted=0`
   - `status=0`

3. 保存用户组授权
   - 删除旧 `admin_group_menu`
   - 插入新 `admin_group_menu`

4. 更新短信配置
   - 多个 key-value 必须一次性提交成功

## 7. 风险点与落地处理

### 7.1 风险：单个 logic 文件失控膨胀

处理：

- 登录模块拆 `login.go / login_sms.go / login_verify.go / logout.go / me.go`
- 用户模块拆 `list.go / add.go / edit.go / delete.go / restore.go / batch_business.go`
- 用户组模块拆 `list.go / add.go / edit.go / delete.go / auth_get.go / auth_save.go`

### 7.2 风险：权限变更后在线用户不生效

处理：

- 不把权限列表作为 token 真值
- 权限在 Redis/DB 中按请求读取
- 用户组授权保存后删除组权限缓存
- 下次请求自动拿到最新权限

### 7.3 风险：被删除员工恢复后用户名冲突

处理：

- 删除时将用户名改为 `原用户名_时间戳`
- 恢复时先去掉后缀
- 若去后缀后的用户名已被占用，直接返回业务冲突，不强行覆盖

### 7.4 风险：短信配置未配置导致登录链路断裂

处理：

- 登录第一步无需读取短信配置
- 仅在实际发送验证码时读取配置
- 配置缺失时明确返回“短信配置未完成，请联系超级管理员”

## 8. 交付结果标准

本项目最终必须达到：

- 所有接口按需求文档落地
- 鉴权、授权、短信验证、软删除、恢复、日志均闭环
- 目录清晰、文件细分，无超大单文件
- 关键逻辑均有注释
- SQL、seed、接口、缓存、日志均有文档可追踪
